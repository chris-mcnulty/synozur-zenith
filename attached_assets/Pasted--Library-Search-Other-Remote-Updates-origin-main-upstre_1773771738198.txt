This is a great architectural discussion. You're describing a significant new data layer — document libraries as first-class entities in Zenith's inventory, sitting below workspaces in the hierarchy. Let me think through the design considerations with you.

Data Model

Each workspace (SharePoint site) can have many document libraries. A document_libraries table would be tenant-owned (like workspaces) and reference the parent workspace. Core properties to capture:

Display name, URL, ID (from Graph/SharePoint)
Sensitivity label (libraries inherit or override from the site)
Item count, storage used
Content types applied
Custom metadata columns
Syntex models deployed
Knowledge agent / Autofill configuration
Last modified, created date
Sync Strategy

This is the key design decision. Three options:

Bundled with tenant sync — fetch libraries for every site during the full sync. Simple, but significantly heavier (could be 5-10x more API calls).

Separate library sync — a dedicated "Library Sync" action, either per-workspace or tenant-wide. More lightweight per-call since it only hits the Lists API. Could run on a different cadence than the full tenant sync.

Hybrid — full tenant sync captures basic library metadata (name, item count, sensitivity label) as a lightweight pass. A deeper "Library Enrichment" sync captures content types, Syntex models, custom columns, and Autofill config on demand or per-workspace.

I'd lean toward option 3. The basic library data is cheap to fetch (/sites/{id}/lists?$filter=list/template eq 'documentLibrary') and could ride along with the existing sync. The heavier content type / Syntex / AI data would be a separate enrichment pass — similar to how we already do property bag enrichment.

UX

Workspace detail view: A "Document Libraries" tab showing all libraries for that site, with sensitivity label, item count, size, content types count
Dedicated library view: A cross-workspace table of all libraries (filterable by tenant, workspace, sensitivity label, content type, etc.)
Future: central definition management for Syntex models, Autofill columns, content types — allowing "deploy at scale" workflows
Central Definitions

This is the forward-looking piece. Tables like content_type_definitions, syntex_model_definitions, autofill_column_definitions would be tenant-owned templates. Zenith would track which libraries have which definitions deployed, enabling gap analysis ("these 50 libraries should have this Syntex model but only 30 do") and bulk deployment.
